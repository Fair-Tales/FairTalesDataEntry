## P1 — HIGH — Batch upload (#84) & orphan reconstruction 100% broken since #217 — **FIXED (715936d)**
`pages/add_books_batch.py:238` and `book_reconstruction.py:388` unpacked **2**
values from `_process_page(...)`, but #217 (20772d7) changed it to a **3-tuple**
(`pages/uploader.py:215`: `bytes_for_extraction, method, rotation_uncertain`).
Result: `ValueError: too many values to unpack` on every page, swallowed by the
per-page isolation `except`, page registered **blank**. Tests missed it because
their `_process_page` mocks returned a 2-tuple (arity drift). Verified directly.
Fix applied: unpack 3 at both sites; mocks corrected to 3-tuple so they now
exercise the real unpack.

## P2 — HIGH — Editing a registered book's TITLE corrupts/overwrites data
`Book.document_id` is title-derived (`data_structures/book.py:332`) but the edit
form leaves the title editable (`book.py:98`, no `disabled=`). On submit
`self.title = _title` write-throughs against the *new* doc id: a unique new title
→ `NotFound` mid-submit (session then writes fields to a phantom doc); a title
colliding with an existing book → the exists-check is skipped for `editing`
books (`book.py:249`) and subsequent author/publisher/theme writes **silently
overwrite the other book**, orphaning the original; S3 folder stays under the old
title so page images 404. `validation.py:234-239` disables its title field and
`scripts/rename_book.py` exists for this — the student form never got the lock.
Fix: disable/skip the title assignment when `self.is_registered`.

## P3 — HIGH — `Book.characters` read-modify-write drops characters under concurrency
`pages/enter_text.py:922-929` and `data_structures/book.py:347-354` rewrite the
whole `characters` list from the in-memory copy loaded at book open. Two tabs /
archivist+validator on one book → last write wins; other session's refs vanish
from the cast. Fix: `firestore.ArrayUnion`/`ArrayRemove`.

## P4 — HIGH — Legacy mixed-case user docs may be locked out / duplicated NOW (needs console audit)
Case-normalization fix (`278a4a5`) lowercased lookups but shipped no migration;
merge `dfa7a40` says "existing mixed-case user docs still need manual cleanup".
For a pre-2026-07-06 user with uppercase in username/email: login fails with the
correct password (`utilities.py:1020`), reset silently no-ops, re-registration
creates a duplicate empty account while their books' `entered_by` point at the
old id. Action: audit `users` for `id != id.lower()` and books with mixed-case
`entered_by`; confirm with Chris whether cleanup was done. NOT code-verifiable.

## P5 — HIGH — Failed character detection re-fires the paid whole-book AI call every rerun
`pages/enter_text.py:1075-1093`: when `_detected_characters` is absent,
`detect_entry` always calls `run_character_detection()`. On failure it shows an
error but leaves `now_entering='detect'` with no failure flag, so every rerun
(Prev/Next, show-original toggle, resize, fragment interaction) fires another
unbounded full-book Anthropic call. Fix: set `_detection_failed`, require an
explicit re-run click.

## MEDIUM — Photo upload & S3
- **Batch & collection flows have NO upload-completion gating** (the #199 bug,
  unfixed here). `pages/add_books_batch.py:375` and `pages/collection_picker.py:436`
  call `fetch_uploaded_photos` from the Detect/Extract button with no
  `uploads_settled` check. Mid-upload "Detect books" reads a partial batch. Fix:
  gate both like the single/append/pages flows.
- **Multi-tab single-book entries share one temp prefix.** `get_upload_session_id("single")`
  recovers the first tab's session id from Firestore (`photo_upload.py:189`), so a
  second tab PUTs into the same `uploads/single/{id}/` → two books interleave.
- **Background job's extracted-title working prefix can collide** between
  concurrent entries of the same title (`background_pipeline.py:487-498`; only
  guarded by `fs.exists(page_1.jpg)`). Use the job-id placeholder folder as the
  working area; reconcile to title at finalize.

## MEDIUM — Firestore write-through & data integrity
- **Silent `None` ref overwrite on submit.** `self.author = _author` resolves a
  display name via `st.session_state['author_dict'].get(name, None)`; a miss
  silently stores `None` (`base_structure.py:27-28`). Same class:
  `Character(book=title)`/`Alias(book=title)` (`enter_text.py:567,586`) →
  `book=None` → later `AttributeError` in `document_id`. Residual seeding hole: a
  book with no illustrator edited with a stale `current_illustrator` seeds+saves
  the WRONG illustrator (`book.py:190-196`); edit path never clears `current_*`.
- **Deleted-ref guard (247ade5) missing at sibling sites:** `pages/user_home.py:147`
  (`author_search`), `book.py:165` (publisher fallback), `alias.py:63`,
  `page_photo_upload.py:31`, `confirm.py:16`.
- **Silent text loss on a page whose Firestore doc is missing.** `enter_text.py:41`
  builds an unregistered `Page(None)`; typed text no-ops and vanishes, no warning
  (validation.py:405 handles this; enter-text doesn't).
- **Lookup-dict loaders crash session init on one malformed doc.** `utilities.py:2682-2714`
  (`load_publisher_dict`/`load_book_dict`/`load_character_dict`) use bare
  `['name']`/`['title']` in a shared `st.cache_resource` → one partial doc raises
  KeyError for every user. Use `.get()` + skip-and-log.

## MEDIUM — Rotation / crop / image
- **enter-text has no S3-miss guard;** one missing page image crashes the page.
  `load_page_image` raises `FileNotFoundError`; `validation.py:352` wraps it but
  enter-text `display_image()` (345/348), crop dialog (217), Enlarge (375) call
  it bare.
- **Correctly-cropped-but-sideways page loses its crop.** Crop-quality gate runs
  before rotation and demands "RIGHT WAY UP", so a good 90/180 crop is rejected
  and stage 2 saves the rotated RAW as `_cropped`, labelled "Auto-corrected".
  Fix: rotate the crop and re-gate once before falling back to raw.
- **`rotation_uncertain` never surfaced in validation & can't be cleared** without
  a full crop-and-rotate save; batch/reconstruction never set it (route through
  `_register_processed_page`).

## MEDIUM — Character / alias / validation
- **Heartbeat lock never released; stale window 2h.** `validation.py:588` writes
  on open + every 30s but expiry is 120 min; Back-to-list / close never clear it,
  so a 10s peek locks the owner out up to 2h. Clear `validation_active_at=None` on
  Back-to-list and Approve; shorten window.
- **No validator mutual exclusion.** `validation.py` never reads the heartbeat;
  two validators edit (last-write-wins) and both Approve; list shows
  `entry_status='started'` books (Approve doesn't re-check status).
- **Possessive stripping mangles titles.** `strip_leading_article` (`utilities.py:1816`)
  → "Her Majesty"→"Majesty", "My Naughty Little Sister"→"Naughty Little Sister".
- **Alias id collisions after stripping dropped silently.** Alias
  `document_id = <book_id>_<name>` (no character); "my mum"→"mum" collides with
  "Mum"; `commit_detected_characters` (`enter_text.py:897`) skips dups silently.

## MEDIUM — Auth / cookies
- **Remember-me cookies survive a password reset** (`cookie_auth.py:321` checks
  only user-exists). Bake a credential fingerprint / `token_version` into the
  payload; compare on restore (also fixes sign-out abandonment race).
- **Reset/confirm email links not URL-encoded** (`utilities.py:1361/1387`);
  `chris+pilot@gmail.com` decodes `+` as space. Use `urllib.parse.urlencode`.
- **`pages/confirm.py` crashes on missing token/user** (unguarded
  `st.query_params.token`, `user_data['is_confirmed']`); non-constant-time compare.
